1、问题描述
汽车销售信息管理系统用于管理汽车4S店或经销商的日常业务数据。系统需要管理的实体包括：
- 轿车信息：编号、型号、颜色、生产厂家、出厂日期、价格
- 员工信息：编号、姓名、性别、年龄、籍贯、学历
- 客户信息：编号、名称、联系方式、地址、业务联系记录
- 销售信息：记录编号、销售日期、汽车编号、型号、颜色、数量、经手人、客户、总价

销售记录关联轿车、员工、客户三个实体，需要实现跨表关联查询和自动填充。

2、功能要求
（1）轿车信息管理：添加、删除、修改、查询（按编号/型号/价格范围）、显示全部
（2）员工信息管理：添加、删除、修改、查询（按编号/姓名/学历）、显示全部
（3）客户信息管理：添加、删除、修改、查询（按编号/名称）、显示全部、追加业务联系记录
（4）销售信息管理：添加（自动关联车辆/员工/客户信息）、删除、查询（按编号/型号/经手人/日期范围）、显示全部
（5）销售报表统计：按型号统计销售数量和金额、按经手人统计销售数量和金额
（6）数据持久化：退出系统时自动保存全部数据到文件，启动时自动加载

前言

本汽车销售信息管理系统基于C语言开发，采用单向链表作为核心数据结构，二进制文件作为数据持久化方式。系统针对汽车销售行业日常业务管理需求，围绕轿车、员工、客户、销售四类核心数据的增删改查场景，实现完整的信息管理功能。

项目采用模块化设计，将系统划分为轿车管理、员工管理、客户管理、销售管理、数据I/O和通用工具六个独立模块，通过头文件声明接口、源文件实现逻辑的方式组织代码。数据层使用链表存储，文件层使用 fwrite/fread 二进制读写，形成内存链表 + 磁盘文件的双重存储机制。开发过程紧扣C语言指针操作、链表遍历、结构体封装、文件I/O等核心知识点，把课本理论转化为可独立运行的实用软件。

系统在设计阶段完善了输入校验逻辑，对编号唯一性、空输入等边界情况做了处理；销售记录添加时自动关联车辆、员工、客户信息，减少重复录入；销售报表支持按型号和按经手人两个维度统计，满足基本的业务分析需求。本次课程设计完整走完需求分析、数据结构设计、模块开发与调试测试全流程，也为后续更复杂的数据管理系统开发积累了实践经验。

一、需求分析

1.1 项目背景
传统汽车销售业务中，车辆库存、销售人员、客户信息和交易记录往往分散在纸质档案或简易电子表格中，查询效率低、数据易丢失、统计工作繁琐。为实现电子化管理，采用C语言开发本系统，集中管理轿车、员工、客户和销售四类数据，完成信息增删改查与报表统计，提升业务管理效率。

1.2 功能需求
1. 轿车管理：添加车辆（编号唯一性检查）、删除（确认后删除）、修改（回车保留原值）、查询（支持按编号精确查询、按型号模糊查询、按价格范围查询）、显示全部车辆列表。
2. 员工管理：添加员工（编号查重）、删除、修改、查询（支持按编号、按姓名模糊、按学历筛选）、显示全部。
3. 客户管理：添加客户（编号自动生成 C0001/C0002...）、删除、修改、查询（支持按编号、按名称模糊、查看含联系记录的完整信息）、显示全部、追加业务联系记录（自动添加日期戳）。
4. 销售管理：添加销售记录（编号自动生成 S0001/S0002...，自动关联车辆/员工/客户并填充冗余字段，自动计算总价）、删除、查询（支持按编号、按车型、按经手人、按日期范围）、显示全部、生成销售报表（按型号统计 + 按经手人统计，同时导出文本文件）。
5. 数据持久化：系统启动时自动从文件加载数据，退出时自动保存全部数据到二进制文件。

1.3 数据需求
1. 轿车结构体（Car）：
   - id: 编号（char[64]，主键唯一）
   - model: 型号（char[64]）
   - color: 颜色（char[64]）
   - manufacturer: 生产厂家（char[64]）
   - date: 出厂日期（char[64]，格式 YYYY-MM-DD）
   - price: 价格（double）
   - next: 链表指针

2. 员工结构体（Employee）：
   - id: 编号（char[64]，主键唯一）
   - name: 姓名（char[64]）
   - gender: 性别（char[4]）
   - age: 年龄（int）
   - hometown: 籍贯（char[64]）
   - education: 学历（enum: 1-高中 2-大专 3-本科 4-硕士 5-博士）
   - next: 链表指针

3. 客户结构体（Customer）：
   - id: 编号（char[64]，自动生成 C0001...）
   - name: 客户名称（char[64]）
   - phone: 联系方式（char[64]）
   - address: 地址（char[256]）
   - contact_log: 业务联系记录（char[256]，格式 [YYYY-MM-DD] 内容; 追加）
   - next: 链表指针

4. 销售结构体（Sale）：
   - id: 记录编号（char[64]，自动生成 S0001...）
   - date: 销售日期（char[64]，格式 YYYY-MM-DD）
   - car_id: 关联车辆编号（char[64]）
   - car_model: 型号（char[64]，冗余存储）
   - color: 颜色（char[64]，冗余存储）
   - quantity: 数量（int）
   - emp_id: 经手人编号（char[64]）
   - emp_name: 经手人姓名（char[64]，冗余存储）
   - cust_id: 客户编号（char[64]）
   - cust_name: 客户名称（char[64]，冗余存储）
   - total_price: 总价（double，= 单价 × 数量）
   - next: 链表指针

5. 实体间关系：
   - Sale.car_id → Car.id（一对多，一辆车可被多次销售）
   - Sale.emp_id → Employee.id（一对多，一个员工可经手多条记录）
   - Sale.cust_id → Customer.id（一对多，一个客户可有多个订单）

1.4 运行环境
- 开发语言：C11
- 编译工具链：GCC + CMake
- 操作系统：Linux（Arch Linux），兼容 Windows
- 构建系统：CMake 3.10+
- 运行方式：命令行终端

三、详细设计

3.1 系统功能模块划分

1. 主控模块（main.c + menu.c）
main.c：程序入口，调用 load_all_data() 加载数据后进入 main_menu() 循环。
menu.c：主菜单显示与分发，包含6个选项（轿车/员工/客户/销售/保存/退出）。退出时自动保存数据。

2. 轿车管理模块（car.c/h）
核心数据结构：Car 单向链表（含尾指针 car_tail 优化尾部追加）
功能函数：
- car_add(): 添加车辆，编号查重，尾插法插入
- car_delete(): 按编号删除，支持确认取消，维护头尾指针
- car_modify(): 按编号修改，支持回车保留原值
- car_query(): 子菜单，支持按编号/型号模糊/价格范围三种查询
- car_list_all(): 显示全部车辆
- car_find(id): 按编号精确查找，返回节点指针或NULL
- car_save/load(): 二进制文件读写

3. 员工管理模块（employee.c/h）
核心数据结构：Employee 单向链表 + Education 枚举
功能函数：
- emp_add(): 添加员工，编号查重，学历枚举输入验证
- emp_delete(): 按编号删除，确认后摘链
- emp_modify(): 按编号修改，支持回车保留原值
- emp_query(): 子菜单，支持按编号/姓名模糊/学历筛选
- emp_list_all(): 显示全部员工
- emp_find(id): 按编号精确查找
- emp_save/load(): 二进制文件读写
- edu_to_str(): 枚举转中文显示

4. 客户管理模块（customer.c/h）
核心数据结构：Customer 单向链表 + 自增编号计数器
功能函数：
- cust_add(): 添加客户，编号自动生成（C0001...），跳过已存在编号
- cust_delete(): 按编号删除
- cust_modify(): 按编号修改（不含联系记录）
- cust_query(): 子菜单，支持按编号/名称模糊/查看完整信息
- cust_list_all(): 显示全部客户
- cust_add_contact_log(): 追加业务联系记录，自动添加日期戳
- cust_find(id): 按编号精确查找
- cust_save/load(): 二进制文件读写，加载时同步更新编号计数器

5. 销售管理模块（sale.c/h）
核心数据结构：Sale 单向链表 + SaleStat 统计结构体
功能函数：
- sale_add(): 添加销售记录，自动关联车辆/员工/客户并填充冗余字段，自动计算总价
- sale_delete(): 按编号删除
- sale_query(): 子菜单，支持按编号/车型/经手人/日期范围四种查询
- sale_list_all(): 显示全部销售记录
- sale_report(): 生成销售报表（按型号+按经手人统计），导出 report.txt
- sale_find(id): 按编号精确查找
- sale_save/load(): 二进制文件读写，加载时同步更新编号计数器

6. 数据I/O模块（data_io.c/h）
- load_all_data(): 启动时调用，按序加载四类数据
- save_all_data(): 退出/手动保存时调用，按序保存四类数据
- ensure_data_dir(): 确保 data/ 目录存在（跨平台 mkdir）

7. 工具模块（utils.c/h）
- clear_screen(): 清屏（Linux clear / Windows cls）
- pause_screen(): 按回车继续
- safe_gets(): 安全读取一行（处理缓冲区残留）
- read_int(): 安全读取整数
- get_current_date(): 获取当前日期字符串（YYYY-MM-DD）

3.2 数据结构详细设计

各结构体定义见 include/ 头文件，核心设计说明：

1. 链表采用尾插法（而非头插法）：每个模块维护 head 和 tail 两个指针，新节点追加到尾部。好处是列表中数据的显示顺序与添加顺序一致，用户体验更好。

2. 删除节点时同时维护 tail 指针：当删除的是尾节点时，将 tail 指向前驱节点，保证 tail 始终有效。

3. 销售记录的冗余存储：car_model、color、emp_name、cust_name 在添加销售记录时从关联表查出后复制保存。这样做的好处是报表查询和显示时不依赖其他模块的数据完整性，即使车辆或员工被删除，历史销售记录仍然完整可读。

4. 客户编号和销售编号自动生成：使用静态计数器 cust_id_counter 和 sale_id_counter，每次自增后格式化输出（snprintf）。加载数据时扫描已有编号更新计数器最大值，避免重启后编号冲突。

3.3 核心函数功能与参数说明

1. 通用查找函数模式
   - Car *car_find(const char *id): 遍历 car_head 链表，strcmp 比较 id，找到返回指针，否则返回 NULL
   - Employee *emp_find(const char *id): 同上
   - Customer *cust_find(const char *id): 同上
   - Sale *sale_find(const char *id): 同上

2. 添加函数模式
   - car_add(): 分配节点→输入编号→查重→输入其余字段→尾插→提示成功
   - emp_add(): 类似，额外处理学历枚举输入验证
   - cust_add(): 编号自动生成→跳过已存在编号→输入其余字段→追加首次联系记录→尾插
   - sale_add(): 编号自动生成→输入日期（默认当日）→关联车辆（自动填充型号/颜色/单价）→输入数量→关联员工→关联客户→计算总价→尾插

3. 删除函数模式
   所有删除函数遵循统一流程：输入编号→遍历查找（记录 prev）→显示记录→确认→摘链（分 prev==NULL 和 prev!=NULL 两种情况）→更新 tail→free→提示

4. 修改函数模式
   所有修改函数遵循统一流程：输入编号→查找→显示当前值→逐字段提示"回车保留原值"→仅当输入非空时才更新

5. 查询函数
   各模块查询均为子菜单循环，支持多种查询方式：
   - car_query: 编号精确 / 型号模糊(strstr) / 价格范围
   - emp_query: 编号精确 / 姓名模糊(strstr) / 学历精确
   - cust_query: 编号精确 / 名称模糊(strstr) / 查看完整信息

5.2 系统各功能使用说明

主菜单提供6个选项：
- 选择 1 进入车辆信息管理：支持增删改查和按型号/价格范围筛选
- 选择 2 进入员工信息管理：支持增删改查和按姓名/学历筛选
- 选择 3 进入客户信息管理：支持增删改查、按名称模糊搜索、追加联系记录

5.3 软件使用注意事项

1. 编号唯一性：添加车辆和员工时需要保证编号不重复，系统会自动检查

3. 数据保存：退出系统时会自动保存数据，但建议定期手动保存（主菜单选项5）防止意外退出
4. 客户编号和销售编号由系统自动生成，无需手动输入
5. 业务联系记录追加有长度限制（256字符），超长将被拒绝
6. 销售报表导出为 data/report.txt 文本文件

6.2 常见bug及解决办法

1. 缓冲区残留问题：使用 scanf 后直接 fgets 会读到残留换行符。解决办法：统一使用 safe_gets() 替代 scanf，内部处理缓冲区清空。

2. 删除头节点后 tail 指针未更新：删除唯一节点时 car_tail 仍指向已释放内存。解决办法：删除时判断 if (car_tail == p) car_tail = prev。

3. 客户编号加载后冲突：重启后 cust_id_counter 为0，新编号与已有编号冲突。解决办法：加载时扫描已有编号，更新计数器为最大值。

4. 跨平台编译问题：Windows 和 Linux 的 clear 命令和 mkdir 接口不同。解决办法：使用 #ifdef _WIN32 条件编译分别处理。


七、总结、附录和参考资料

7.1 项目开发总结

本次课程设计完成了一个基于C语言的汽车销售信息管理系统。系统涵盖了链表数据结构的核心操作（创建、插入、删除、遍历查找），文件I/O的二进制读写，以及模块化程序设计思想的实际应用。

在开发过程中遇到的主要挑战：
1. 链表操作的边界条件：头节点删除、尾节点删除、空链表插入等特殊情况需要分别处理，通过维护 head 和 tail 双指针简化了逻辑。
2. 跨模块数据关联：销售记录需要引用车辆、员工、客户的编号并自动填充信息，通过暴露各模块的 _find() 函数实现跨模块查询。
3. 数据持久化设计：选择二进制文件而非文本文件，因为结构体逐条 fwrite/fread 实现简单且效率较高；但需要在保存时将 next 指针置NULL避免序列化指针地址。

收获与不足：
- 加深了对链表操作的理解，掌握了尾插法、前驱节点追踪、尾指针维护等技巧
- 学会了模块化设计的接口分离方法（头文件声明 + 源文件实现）
- 不足之处：未实现排序功能、未支持数据导出为可读文本格式（仅报表支持）、无输入数据的合法性深度校验（如价格不能为负）
